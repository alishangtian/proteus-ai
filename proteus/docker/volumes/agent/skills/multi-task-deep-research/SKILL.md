# 🎯 多任务深度研究技能：企业级智能研究协调系统

## 🌟 技能核心价值：为什么选择多任务深度研究？

多任务深度研究技能是**planning-with-files模式**与**深度研究方法论**的融合升级，专为复杂、多维度、需要并行处理的研究任务而设计。与单任务研究相比，本技能提供：

### 🔍 多任务 vs. 单任务研究：关键区别

| 维度 | 单任务深度研究 | 多任务深度研究 |
|------|---------------|--------------|
| **任务规模** | 单一研究问题 | 多个相关研究子任务 |
| **协调机制** | 线性执行 | 并行执行+进度监控 |
| **输出结构** | 单一报告 | 主报告+子任务报告 |
| **资源利用** | 串行资源分配 | 智能资源调度 |
| **风险控制** | 单一失败点 | 容错与任务隔离 |
| **价值产出** | 深度但单一 | 广度+深度综合 |

### 🏆 技能解决的四大核心问题

1. **复杂问题分解** - 将宏大研究问题拆解为可管理的子任务
2. **并行执行协调** - 智能调度和监控多个子任务进度
3. **结果整合挑战** - 系统化整合分散的研究发现
4. **质量控制统一** - 确保所有子任务达到一致质量标准

## 🚀 快速开始：五分钟上手指南

### 📋 第一步：识别适用场景

**使用本技能当您需要：**
- 同时研究一个主题的多个方面（如技术、市场、政策、应用）
- 协调团队或AI代理并行执行相关研究任务
- 管理长期、复杂的研究项目
- 生成包含多个维度的综合研究报告

**示例触发语句：**
- "请对[主题]进行多维度深度研究，我需要技术、市场、政策等全面分析..."
- "为[决策]提供多视角的综合研究报告，包含..."
- "协调多个研究任务，分析[复杂问题]的不同方面..."
- "管理一个包含多个子任务的研究项目..."
- "生成包含技术趋势、市场分析、风险评估的综合报告..."

### 🛠️ 第二步：初始化多任务研究

```python
# 多任务研究启动代码模板
multi_task_config = {
    "主题": "[您要研究的主题]",
    "研究类型": "[选择: 技术评估/市场分析/战略规划/风险评估/学术研究]",
    "输出格式": "[选择: 执行摘要/详细报告/演示文稿]",

}
```

### 📊 第三步：选择任务协调模式

根据研究复杂度选择协调模式：

| 协调模式 | 适用场景 | 核心特点 |
|----------|----------|----------|
| **完全并行** | 子任务独立无依赖 | 同时启动所有任务，最大并发度 |
| **阶段依赖** | 任务有先后顺序 | 按阶段顺序执行，确保依赖满足 |
| **混合模式** | 部分任务有依赖 | 组合并行和顺序执行，优化效率 |

1. 初始化项目时设置`auto_start_subtasks=False`
2. 手动进入每个子任务目录执行研究
3. 使用`monitor_tasks.py`跟踪进度
4. 使用`integrate_results.py`整合结果

详细步骤参见: [LOCAL_EXECUTION_GUIDE.md](./LOCAL_EXECUTION_GUIDE.md)

```bash
python scripts/local_research_runner.py /app/data/tasks/您的项目 "子任务名称"
```

# 🔧 技能工作机制：多任务研究框架

## 🔄 五阶段智能研究循环

### 阶段1：主任务定义与分解 (15-20%)
**目标**：定义总体研究目标，拆解为可管理的子任务
**关键活动**：
- 澄清研究主题和范围
- 识别关键研究维度和子任务
- 定义子任务依赖关系和优先级
- 创建主任务规划文件

**输出**：主任务计划（master_task_plan.md）、任务配置（task_config.json）

### 阶段2：子任务初始化与规划 (10-15%)
**目标**：为每个子任务创建独立的规划文件和工作目录
**关键活动**：
- 为每个子任务创建独立目录
- 初始化子任务规划文件（task_plan.md, findings.md, progress.md）
- 分配资源和时间预算
- 设定子任务质量标准和验收条件

**输出**：子任务目录结构、子任务规划文件

### 阶段3：并行执行与进度监控 (30-40%)
**目标**：协调多个子任务的并行执行，实时监控进度
**关键活动**：
- **智能调度系统**：
  - 基于依赖关系的任务排序
  - 资源冲突检测和解决
  - 进度滞后预警和调整

- **实时监控机制**：
  - 子任务进度跟踪（progress.md更新）
  - 错误和异常检测
  - 质量检查点执行

### 阶段4：结果收集与初步整合 (15-20%)
**目标**：收集所有子任务结果，进行初步整合
**关键活动**：
- **结果收集**：
  - 提取子任务关键发现（findings.md）
  - 验证子任务完成质量
  - 识别冲突或矛盾的结果

- **初步整合**：
  - 合并相关发现到主发现文件（master_findings.md）
  - 识别跨子任务的模式和趋势
  - 标记需要进一步验证的发现

### 阶段5：综合分析与报告生成 (15-20%)
**目标**：生成综合研究报告，提供战略洞察
**关键活动**：
- **综合分析**：
  - 跨维度趋势识别
  - 矛盾解决和共识建立
  - 优先级排序和风险评估

- **报告生成**：
  - 创建执行摘要（给决策者）
  - 生成详细综合报告（给执行者）
  - 准备演示材料（给利益相关者）

# 🏗️ 文件系统架构：基于planning-with-files模式

## 📁 标准目录结构

```
项目根目录/
├── master_task_plan.md          # 主任务规划
├── master_findings.md           # 主研究发现汇总
├── master_progress.md           # 主任务进度跟踪
├── task_config.json             # 任务配置和状态
├── sub_tasks/                   # 子任务目录
│   ├── [子任务1]/
│   │   ├── task_plan.md         # 子任务规划
│   │   ├── findings.md          # 子任务发现
│   │   └── progress.md          # 子任务进度
│   ├── [子任务2]/
│   │   ├── task_plan.md
│   │   ├── findings.md
│   │   └── progress.md
│   └── ...
├── reports/                     # 报告输出目录
│   ├── progress_report_[时间戳].md
│   └── final_report_[时间戳].md
└── data/                        # 研究数据目录
    ├── sources/                 # 收集的原始资料
    └── analysis/                # 分析中间结果
```

## 📄 核心文件说明

### 1. master_task_plan.md (主任务规划)
- **目的**：定义总体研究目标、范围、子任务分解、时间规划
- **内容结构**：
  - 总体目标和研究问题
  - 子任务分解和描述
  - 时间规划和里程碑
  - 资源分配和依赖关系
  - 质量标准和验收条件

### 2. master_findings.md (主研究发现)
- **目的**：汇总所有子任务的关键发现，进行跨任务整合
- **内容结构**：
  - 执行摘要和核心结论
  - 按主题分类的发现汇总
  - 跨子任务的模式识别
  - 矛盾发现和解决建议
  - 未解答问题和未来研究方向

### 3. master_progress.md (主任务进度)
- **目的**：跟踪整体研究进度，监控子任务状态
- **内容结构**：
  - 总体进度百分比和预计完成时间
  - 子任务状态跟踪表
  - 风险和问题日志
  - 资源使用情况
  - 下一次检查点计划

### 4. task_config.json (任务配置)
- **目的**：机器可读的任务配置和状态跟踪
- **内容结构**：
  - 任务元数据（名称、描述、创建时间）
  - 子任务列表和状态
  - 监控配置（检查间隔、通知设置）
  - API配置（工具访问权限）
  - 进度统计和性能指标

### 5. 子任务文件（task_plan.md, findings.md, progress.md）
- **目的**：每个子任务的独立规划、发现和进度跟踪
- **遵循标准**：继承planning-with-files模式的核心原则

## 🚀 子任务自动启动系统

多任务深度研究技能现在集成了任务启动API，支持在子任务创建完成后自动启动独立的研究任务。每个子任务将作为一个独立的AI研究任务运行，充分利用并行计算资源。

### 🔌 API集成配置

任务启动系统使用以下API配置（可在`task_config.json`的`api_config`部分自定义）：

```json
{
  "endpoint": "https://127.0.0.1/task",
  "auth_token": "您的认证令牌",
  "model_name": "deepseek-reasoner",
  "itecount": 200,
  "conversation_round": 5,
  "tool_choices": ["serper_search", "web_crawler", "python_execute"],
  "selected_skills": ["planning-with-files", "deep-research", "multi-task-deep-research"]
}
```

### 🛠️ 启动方式

#### 方式一：初始化时自动启动
在初始化多任务研究时，设置`auto_start_subtasks=True`参数：

```python
init_multi_task_research(
    task_name="AI趋势研究",
    task_description="全面研究AI技术发展趋势",
    subtasks=[...],
    auto_start_subtasks=True
)
```

#### 方式二：手动启动子任务
使用`start_subtasks.py`脚本手动启动：

```bash
# 启动所有未开始的子任务
python /app/.proteus/skills/multi-task-deep-research/scripts/start_subtasks.py /app/data/tasks/您的任务目录

# 启动特定子任务
python /app/.proteus/skills/multi-task-deep-research/scripts/start_subtasks.py /app/data/tasks/您的任务目录 --specific "子任务A" "子任务B"
```

#### 方式三：通过Python代码启动
```python
from scripts.start_subtasks import start_subtasks
success = start_subtasks("/app/data/tasks/您的任务目录", start_all=True)
```

### 📡 API请求参数说明

每个子任务启动时，系统会发送包含以下参数的POST请求：

| 参数 | 说明 | 示例值 |
|------|------|--------|
| `query` | 研究查询语句 | "分析2025年AI教育技术市场趋势" |
| `workspace_path` | 子任务工作目录 | "/app/data/tasks/AI研究/sub_tasks/市场分析" |
| `model_name` | 使用的AI模型 | "deepseek-reasoner" |
| `itecount` | 迭代次数 | 200 |
| `conversation_round` | 对话轮次 | 5 |
| `tool_choices` | 可用工具列表 | ["serper_search", "web_crawler", "python_execute"] |
| `selected_skills` | 启用的技能 | ["planning-with-files"] |
| `stream` | 流式输出 | true |

### 🔄 状态跟踪与错误处理

系统自动跟踪每个子任务的状态：
- **pending**：等待启动
- **running**：已启动并运行中
- **completed**：已完成
- **error**：启动失败或执行错误

启动失败时，系统会记录错误次数并允许重试。所有状态信息保存在`task_config.json`中。

### ⚠️ 注意事项

1. **认证令牌**：确保`auth_token`正确，对应有效的Bearer令牌
2. **网络连接**：确保可以访问API端点（通常为本地服务）
3. **资源限制**：同时启动过多子任务可能导致资源竞争
4. **错误处理**：建议监控启动状态并及时处理失败任务

### 📊 性能优化建议

- **分批启动**：对于大量子任务，建议分批启动（每次3-5个）
- **进度监控**：结合`monitor_tasks.py`实时监控子任务进度
- **结果整合**：使用`integrate_results.py`定期整合子任务结果


## 🔗 依赖关系管理

多任务深度研究技能现在支持复杂的子任务依赖关系管理，确保有依赖关系的任务按正确顺序执行。

### 📋 依赖关系定义

在创建子任务时，可以指定该任务依赖的其他子任务：

```python
subtasks = [
    {
        "name": "市场分析",
        "description": "分析目标市场规模和趋势",
        "dependencies": []  # 无依赖
    },
    {
        "name": "竞品分析", 
        "description": "分析主要竞争对手",
        "dependencies": ["市场分析"]  # 依赖市场分析
    },
    {
        "name": "战略建议",
        "description": "基于分析提出战略建议",
```

### 🔄 依赖关系执行模式

系统提供两种依赖关系执行模式：

#### 模式一：顺序启动模式 (默认)
- 按依赖关系拓扑排序确定执行顺序
- 一次只启动一个任务，不考虑前一个任务是否完成
- 适用于任务独立但需要特定输入顺序的情况

```bash
python /app/.proteus/skills/multi-task-deep-research/scripts/start_subtasks.py /app/data/tasks/任务目录
# 或
python /app/.proteus/skills/multi-task-deep-research/scripts/start_subtasks.py /app/data/tasks/任务目录 --sequential
```

#### 模式二：顺序等待模式
- 按依赖关系拓扑排序确定执行顺序
- 每个任务完成后才启动下一个任务
- 确保前一个任务的输出可作为下一个任务的输入

```bash
python /app/.proteus/skills/multi-task-deep-research/scripts/start_subtasks.py /app/data/tasks/任务目录 --wait
```

### 🧮 拓扑排序算法

系统使用Kahn算法进行拓扑排序，确保：
1. 无依赖的任务优先执行
2. 有依赖的任务在其所有依赖完成后才可执行
3. 检测循环依赖并给出警告

### 📊 依赖关系验证

在任务初始化时，系统会自动验证：
- 依赖的任务是否存在
- 是否存在循环依赖
- 依赖关系是否形成有效的DAG（有向无环图）

### ⚠️ 循环依赖处理

如果检测到循环依赖：
1. 系统会给出明确警告
2. 尝试进行部分排序
3. 剩余任务（形成循环的部分）将按原始顺序执行

# 🛠️ 实用技巧与最佳实践

## 💡 提高多任务研究效率的技巧

### 1. 智能任务分解
- 遵循MECE原则（相互独立，完全穷尽）
- 每个子任务应聚焦一个核心维度
- 保持子任务规模适中（2-8小时研究时间）
- 明确子任务之间的接口和依赖

### 2. 高效协调策略
- **完全并行模式**：适合无依赖、资源充足的情况
- **阶段依赖模式**：适合有严格先后顺序的任务
- **混合模式**：平衡效率和依赖的折中方案
- **动态调整**：根据执行情况灵活调整协调策略

### 3. 进度监控最佳实践
- 设置明确的里程碑和检查点
- 定期生成进度报告（建议每2小时）
- 建立早期预警机制（进度滞后>20%时报警）
- 保持主进度文件的实时更新

### 4. 结果整合策略
- 建立统一的知识分类体系
- 使用标准化模板提取子任务发现
- 定期进行跨任务发现整合（建议每完成2个子任务）
- 建立矛盾解决机制（第三方验证、专家咨询）

## ⚠️ 常见陷阱与避免方法

### 陷阱1：过度分解
**问题**：子任务划分过细，增加协调成本
**解决方法**：保持子任务规模适中，避免<1小时的任务

### 陷阱2：依赖管理混乱
**问题**：依赖关系未明确定义，导致执行阻塞
**解决方法**：清晰定义依赖关系，建立依赖可视化图表

### 陷阱3：进度同步失败
**问题**：子任务进度差异过大，影响整合
**解决方法**：设置进度同步检查点，动态调整资源分配

### 陷阱4：整合质量不足
**问题**：简单堆砌子任务结果，缺乏深度整合
**解决方法**：预留专门整合时间，使用系统化整合框架
